## Key Question: What does FunctionState do?

`FunctionState` in V8's parser is a stack-based mechanism that saves and restores the current parsing state when entering/exiting a function. It's a RAII pattern - the constructor saves the current state and sets up the new function's state, and the destructor restores the previous state.

The critical question is: **Does `FunctionState` own the `DeclarationScope`?**

### FunctionState's Purpose

`FunctionState` is designed to manage parser state transitions. When you enter a function:
1. The constructor saves the current `function_state_` and `scope_`
2. Sets `function_state_` and `scope_` to point to the new function's state/scope
3. When the destructor runs, it restores the previous `function_state_` and `scope_`

**It does NOT own or manage the lifetime of the `DeclarationScope` object.** It merely holds pointers to it for the duration of parsing the function body.

### DeclarationScope Memory Lifecycle

`DeclarationScope` objects in V8's parser are allocated through `NewFunctionScope()`, which typically uses a `Zone` allocator. Zone-allocated objects have the following properties:

1. They live for the duration of the Zone's lifetime
2. They're never individually freed - the entire Zone is deallocated at once
3. They're safe to use as long as the Zone is alive

The `NewFunctionScope()` method creates a `DeclarationScope` using the parser's zone allocator. This means:

- The scope is allocated on a Zone
- Zone allocations persist until the Zone is destroyed
- The Zone's lifetime is typically tied to the parser's lifetime
- `FunctionState` does NOT free the scope - it only restores the parser's state pointers

# Verification Analysis: DefaultConstructor "UAF"

## Verdict: **FALSE POSITIVE**

## Detailed Analysis

### Question 1: Is this actually a UAF?

**No.** The fundamental misunderstanding is about V8's scope memory model. `DeclarationScope` objects are **zone-allocated**, not heap-allocated. Zone memory is never individually freed — it's bulk-released when the entire Zone is destroyed (which happens after parsing completes). The pointer remains valid for the full duration of parsing.

### Question 2: Does FunctionState destructor free function_scope?

**No.** `FunctionState` is a **state manager, not a resource owner.** Its destructor simply restores the parser's state pointers:

```cpp
~FunctionState() {
  function_state_ = previous_function_state_;
  scope_ = previous_scope_;
  // That's it. No free, no delete, no ownership transfer.
}
```

The destructor reverts `parser->function_state_` and `parser->scope_` to their prior values. It has **zero effect** on the `DeclarationScope` object's lifetime.

### Question 3: Is expected_property_count access safe?

**Yes.** Since `DeclarationScope` is zone-allocated and persists for the entire parsing session, accessing any member (like `expected_property_count`) after the `FunctionState` block ends is perfectly safe. The scope accumulated data during parsing within the block, and that data is now read to construct the `FunctionLiteral`.

### Question 4: What's the actual memory lifecycle of DeclarationScope?

```
1. NewFunctionScope(kind)     → Zone-allocated, pointer on stack
2. FunctionState constructor  → Saves old state, sets scope as "current"
3. { body parsing }           → Scope accumulates declarations
4. FunctionState destructor   → Restores previous state pointer ONLY
5. NewFunctionLiteral(...)    → Reads scope data, creates AST node
6. ... parsing continues ...
7. Zone destruction           → Bulk free of ALL zone allocations
```

The scope outlives the `FunctionState` RAII block by design.

## Why This Is a False Positive

**This is V8's standard, pervasive pattern.** The same `FunctionState` RAII + post-block `NewFunctionLiteral` pattern is used in `ParseFunctionLiteral`, `ParseFunctionDeclaration`, `ParseClassLiteral`, and every other function-parsing path. If this were a real UAF, V8 would crash on nearly every function declaration — a bug that would be impossible to miss.

The researcher conflated **state management** (what `FunctionState` does) with **lifecycle management** (what Zone allocation does). The RAII scope guard restores the parser's "current scope" pointer; it does not destroy the scope object itself.
